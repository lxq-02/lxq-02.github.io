第一章：软件生命周期、软件生命周期模型（大点）、每一种开发模型和开发方法的对应关系

  • 软件生命周期：（七个阶段）

    • 软件生命周期是指一个软件从提出开发要求直到该软件报废为止的整个时期，分为可行性分析和项目开发计划、需求分析、概要设计、详细设计、编码、测试和维护7个阶段。
    • 可行性分析和项目开发计划（数据流图）
    • 需求分析（流程图）
    • 概要设计（映射成软件结构图、功能模块图）（任务：确定系统的结构）
    • 详细设计
    • 编码
    • 测试（单元测试、集成测试、确认测试、系统测试、验收测试）
    • 维护

  • 软件生命周期模型（大点）、每一种开发模型和开发方法的对应关系

  • 软件生命周期模型是描述软件开发过程中各种活动如何执行的模型。

    1. 瀑布模型——结构化方法、Jackson方法
    2. 增量模型——结构化方法、Jackson方法
    3. 螺旋模型——结构化方法
    4. 喷泉模型——面向对象的开发方法
    5. 基于知识的模型——结构化方法
    6. 变换模型——形式化的维也纳方法
    7. 统一过程——面向对象的开发方法

    结构化方法可使用瀑布模型、增量模型和螺旋模型、基于知识的模型

    Jackson方法可使用瀑布模型、增量模型

    面向对象的开发方法可使用喷泉模型和统一过程

    形式化的维也纳方法只能用变换模型进行开发

第三章：耦合性和内聚性（大点、定义、每一种的特点强弱）、软件结构准则（5条）、算法描述（4个工具）、模块的属性

  • 耦合性（也称块间联系，指软件系统结构中各模块间相互联系紧密程度的一种度量

    1. 无直接耦合：指两个模块之间没有直接的联系
    2. 数据耦合：两个模块之间有调用关系，传递的是简单的数据值（最想要）
    3. 标记耦合：指两个模块之间传递的是数据结构
    4. 控制耦合：指一个模块调用另一个模块时，传递的是控制变量（如开关、标识等）
    5. 公共耦合：指通过一个公告数据环境相互左右的那些模块间的耦合（使用全局数据变量或公用的数据区）
    6. 内容耦合：最高程度的耦合，也是最差的耦合。模块之间有公用的部分

  • 内聚性：也称为块内联系，指模块的功能强度的度量，即一个模块内部各个元素彼此结合的紧密程度的度量

    1. 偶然内聚：指一个模块内的各处理元素之间没有任何联系
    2. 逻辑内聚：指模块内执行几个逻辑上相似的功能，通过参数确定该模块完成哪一步的功能
    3. 时间内聚：把需要同时执行的动作组合在一起形成的模块。
    4. 过程内聚：
    5. 通信内聚：指模块内所有处理元素都在同一个数据结构上操作，或者指各处理使用相同的输入数据或者产生相同的输出数据
    6. 信息内聚：使用相同的数据结构（几个加工同时引用一个共同的数据）
    7. 功能内聚：指模块内所有元素共同完成一个功能，缺一不可。

    1. 接口：指模块的输入与输出

  • 白盒测试（大点、定义、3方法掌握两个循环不用掌握）、

    1. 逻辑覆盖（先画图）

      1. 语句覆盖：使程序中每个语句至少执行一次。（真假分支中的语句）

      2. 判定覆盖（分支覆盖）：使每个判定的真假分支都至少执行一次。

      3. 条件覆盖：使每个判定的每个条件的可能取值至少执行一次。

      4. 判定/条件覆盖：选取足够多的测试用例，使判断中的每个条件的所有可能取值至少执行一次，同时每个判断本身的所有可能判断结果至少执行一次。

      5. 条件组合覆盖：所有可能的条件取值组合至少执行一次

      6. 路径覆盖：覆盖每一个可能的路径

  • ​ 系统测试（幻灯片一页、每种类型）

    1. 恢复测试：以不同的方式强使软件出现故障，检测软件能否恰当地完成恢复
      自动恢复：检测重新初始化、监测点设置、数据恢复、重新启动等是否正确
      人工干预恢复：检测平均恢复时间是否在允许范围内
    2. 安全性测试：设计测试用例，突破软件安全机构的安全保密措施，检验系统预防机制的漏洞
    3. 强度测试：设计测试用例，检验系统能i最高能达到的实际限度，让系统处于资源的异常数量、异常频率、异常批量的条件下测试系统的承受能力。一般比平常限度高5-10倍的限度做测试用例
    4. 性能测试：设计测试用例，并记录软件运行性能，与性能要求比较，检验是否达到性能要求规格。

  • ​ 测试用例的定义（
    测试用例是一组输入和期待的结果，它根据引起故障和检查的目的来使用组件。

  • ​ 软件错误的类型（有哪几种）

    1. 功能错(需求分析错误)
    2. 软件结构错
    3. 数据错
    4. 编码错
    5. 软件集成错
    6. 测试定义与测试执行错误

  • ​ 软件可测试性可具备的特性（课件上好像没有）

    1. 可操作性
    2. 可观察性
    3. 可控制性
    4. 可分解性
    5. 稳定性
    6. 简单易理解

  • ​ 测试组件

    • 是系统中可以孤立进行测试的部分，一个组件可以是对象，一组对象，一个或多个子系统。

  • ​ 调试时修改错误的原则

    1. 注意错误的群集现象、在错误近邻检查
    2. 找到错误的本质并修改
    3. 采用回归测试，避免因修改而引起的新错误
  • 第六章：
  • 维护的副作用（数据文档什么什么三个，每一个副作用有一页，有啥内容）

    1. 编码副作用

      1. 删除或修改一个子程序、一个标号和一个标识符
      2. 改变程序代码的时序关系、改变占用存储的大小、改变逻辑运算符
      3. 修改文件的打开或关闭
      4. 改进程序的执行效率
      5. 把设计上的改变翻译成代码的改变
      6. 为边界条件的逻辑测试做出改变

    2. 数据副作用

      1. 重新定义局部或全局的常量、记录或文件格式
      2. 增加或减少一个数组或高层数据结构的大小
      3. 修改全局或公共数据
      4. 重新初始化控制标志或指针
      5. 重新排列输入、输出或子程序的参数

    3. 文档副作用

      1. 修改交互输入的顺序或格式，没有正确的记入文档中
      2. 过时的文档内容、索引和文本可能造成冲突等
